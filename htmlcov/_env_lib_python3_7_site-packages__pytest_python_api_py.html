254 def __eq__(self, actual): 

255 """ 

256 Return true if the given value is equal to the expected value within 

257 the pre-specified tolerance. 

258 """ 

259 if _is_numpy_array(actual): 

260 # Call ``__eq__()`` manually to prevent infinite-recursion with 

261 # numpy<1.13. See #3748. 

262 return all(self.__eq__(a) for a in actual.flat) 

263 

264 # Short-circuit exact equality. 

265 if actual == self.expected: 

266 return True 

267 

268 # Allow the user to control whether NaNs are considered equal to each 

269 # other or not. The abs() calls are for compatibility with complex 

270 # numbers. 

271 if math.isnan(abs(self.expected)):

275 # if there's a relative tolerance, it will be infinite and infinity 

276 # will seem approximately equal to everything. The equal-to-itself 

277 # case would have been short circuited above, so here we can just 

278 # return false if the expected value is infinite. The abs() call is 

279 # for compatibility with complex numbers. 

280 if math.isinf(abs(self.expected)): 

281 return False

300 # either None or a value specified by the user.

308 raise ValueError("absolute tolerance can't be NaN.")

311 # just return the absolute tolerance. 

312 if self.rel is None:

317 # either None or a value specified by the user. This is done after 

318 # we've made sure the user didn't ask for an absolute tolerance only, 

319 # because we don't want to raise errors about the relative tolerance if 

320 # we aren't even going to use it.

328 )

330 raise ValueError("relative tolerance can't be NaN.")

332 # Return the larger of the relative and absolute tolerances. 

333 return max(relative_tolerance, absolute_tolerance)

345def approx(expected, rel=None, abs=None, nan_ok=False): 

346 """ 

347 Assert that two numbers (or two sets of numbers) are equal to each other 

348 within some tolerance. 

349 

350 Due to the `intricacies of floating-point arithmetic`__, numbers that we 

351 would intuitively expect to be equal are not always so:: 

352 

353 >>> 0.1 + 0.2 == 0.3 

354 False 

355 

356 __ https://docs.python.org/3/tutorial/floatingpoint.html 

357 

358 This problem is commonly encountered when writing tests, e.g. when making 

359 sure that floating-point values are what you expect them to be. One way to 

360 deal with this problem is to assert that two floating-point numbers are 

361 equal to within some appropriate tolerance:: 

362 

363 >>> abs((0.1 + 0.2) - 0.3) < 1e-6 

364 True 

365 

366 However, comparisons like this are tedious to write and difficult to 

367 understand. Furthermore, absolute comparisons like the one above are 

368 usually discouraged because there's no tolerance that works well for all 

369 situations. ``1e-6`` is good for numbers around ``1``, but too small for 

370 very big numbers and too big for very small ones. It's better to express 

371 the tolerance as a fraction of the expected value, but relative comparisons 

372 like that are even more difficult to write correctly and concisely. 

373 

374 The ``approx`` class performs floating-point comparisons using a syntax 

375 that's as intuitive as possible:: 

376 

377 >>> from pytest import approx 

378 >>> 0.1 + 0.2 == approx(0.3) 

379 True 

380 

381 The same syntax also works for sequences of numbers:: 

382 

383 >>> (0.1 + 0.2, 0.2 + 0.4) == approx((0.3, 0.6)) 

384 True 

385 

386 Dictionary *values*:: 

387 

388 >>> {'a': 0.1 + 0.2, 'b': 0.2 + 0.4} == approx({'a': 0.3, 'b': 0.6}) 

389 True 

390 

391 ``numpy`` arrays:: 

392 

393 >>> import numpy as np # doctest: +SKIP 

394 >>> np.array([0.1, 0.2]) + np.array([0.2, 0.4]) == approx(np.array([0.3, 0.6])) # doctest: +SKIP 

395 True 

396 

397 And for a ``numpy`` array against a scalar:: 

398 

399 >>> import numpy as np # doctest: +SKIP 

400 >>> np.array([0.1, 0.2]) + np.array([0.2, 0.1]) == approx(0.3) # doctest: +SKIP 

401 True 

402 

403 By default, ``approx`` considers numbers within a relative tolerance of 

404 ``1e-6`` (i.e. one part in a million) of its expected value to be equal. 

405 This treatment would lead to surprising results if the expected value was 

406 ``0.0``, because nothing but ``0.0`` itself is relatively close to ``0.0``. 

407 To handle this case less surprisingly, ``approx`` also considers numbers 

408 within an absolute tolerance of ``1e-12`` of its expected value to be 

409 equal. Infinity and NaN are special cases. Infinity is only considered 

410 equal to itself, regardless of the relative tolerance. NaN is not 

411 considered equal to anything by default, but you can make it be equal to 

412 itself by setting the ``nan_ok`` argument to True. (This is meant to 

413 facilitate comparing arrays that use NaN to mean "no data".) 

414 

415 Both the relative and absolute tolerances can be changed by passing 

416 arguments to the ``approx`` constructor:: 

417 

418 >>> 1.0001 == approx(1) 

419 False 

420 >>> 1.0001 == approx(1, rel=1e-3) 

421 True 

422 >>> 1.0001 == approx(1, abs=1e-3) 

423 True 

424 

425 If you specify ``abs`` but not ``rel``, the comparison will not consider 

426 the relative tolerance at all. In other words, two numbers that are within 

427 the default relative tolerance of ``1e-6`` will still be considered unequal 

428 if they exceed the specified absolute tolerance. If you specify both 

429 ``abs`` and ``rel``, the numbers will be considered equal if either 

430 tolerance is met:: 

431 

432 >>> 1 + 1e-8 == approx(1) 

433 True 

434 >>> 1 + 1e-8 == approx(1, abs=1e-12) 

435 False 

436 >>> 1 + 1e-8 == approx(1, rel=1e-6, abs=1e-12) 

437 True 

438 

439 If you're thinking about using ``approx``, then you might want to know how 

440 it compares to other good ways of comparing floating-point numbers. All of 

441 these algorithms are based on relative and absolute tolerances and should 

442 agree for the most part, but they do have meaningful differences: 

443 

444 - ``math.isclose(a, b, rel_tol=1e-9, abs_tol=0.0)``: True if the relative 

445 tolerance is met w.r.t. either ``a`` or ``b`` or if the absolute 

446 tolerance is met. Because the relative tolerance is calculated w.r.t. 

447 both ``a`` and ``b``, this test is symmetric (i.e. neither ``a`` nor 

448 ``b`` is a "reference value"). You have to specify an absolute tolerance 

449 if you want to compare to ``0.0`` because there is no tolerance by 

450 default. Only available in python>=3.5. `More information...`__ 

451 

452 __ https://docs.python.org/3/library/math.html#math.isclose 

453 

454 - ``numpy.isclose(a, b, rtol=1e-5, atol=1e-8)``: True if the difference 

455 between ``a`` and ``b`` is less that the sum of the relative tolerance 

456 w.r.t. ``b`` and the absolute tolerance. Because the relative tolerance 

457 is only calculated w.r.t. ``b``, this test is asymmetric and you can 

458 think of ``b`` as the reference value. Support for comparing sequences 

459 is provided by ``numpy.allclose``. `More information...`__ 

460 

461 __ http://docs.scipy.org/doc/numpy-1.10.0/reference/generated/numpy.isclose.html 

462 

463 - ``unittest.TestCase.assertAlmostEqual(a, b)``: True if ``a`` and ``b`` 

464 are within an absolute tolerance of ``1e-7``. No relative tolerance is 

465 considered and the absolute tolerance cannot be changed, so this function 

466 is not appropriate for very large or very small numbers. Also, it's only 

467 available in subclasses of ``unittest.TestCase`` and it's ugly because it 

468 doesn't follow PEP8. `More information...`__ 

469 

470 __ https://docs.python.org/3/library/unittest.html#unittest.TestCase.assertAlmostEqual 

471 

472 - ``a == pytest.approx(b, rel=1e-6, abs=1e-12)``: True if the relative 

473 tolerance is met w.r.t. ``b`` or if the absolute tolerance is met. 

474 Because the relative tolerance is only calculated w.r.t. ``b``, this test 

475 is asymmetric and you can think of ``b`` as the reference value. In the 

476 special case that you explicitly specify an absolute tolerance but not a 

477 relative tolerance, only the absolute tolerance is considered. 

478 

479 .. warning:: 

480 

481 .. versionchanged:: 3.2 

482 

483 In order to avoid inconsistent behavior, ``TypeError`` is 

484 raised for ``>``, ``>=``, ``<`` and ``<=`` comparisons. 

485 The example below illustrates the problem:: 

486 

487 assert approx(0.1) > 0.1 + 1e-10 # calls approx(0.1).__gt__(0.1 + 1e-10) 

488 assert 0.1 + 1e-10 > approx(0.1) # calls approx(0.1).__lt__(0.1 + 1e-10) 

489 

490 In the second example one expects ``approx(0.1).__le__(0.1 + 1e-10)`` 

491 to be called. But instead, ``approx(0.1).__lt__(0.1 + 1e-10)`` is used to 

492 comparison. This is because the call hierarchy of rich comparisons 

493 follows a fixed behavior. `More information...`__ 

494 

495 __ https://docs.python.org/3/reference/datamodel.html#object.__ge__ 

496 """ 

497 

498 # Delegate the comparison to a class that knows how to deal with the type 

499 # of the expected value (e.g. int, float, list, dict, numpy.array, etc). 

500 # 

501 # The primary responsibility of these classes is to implement ``__eq__()`` 

502 # and ``__repr__()``. The former is used to actually check if some 

503 # "actual" value is equivalent to the given expected value within the 

504 # allowed tolerance. The latter is used to show the user the expected 

505 # value and tolerance, in the case that a test failed. 

506 # 

507 # The actual logic for making approximate comparisons can be found in 

508 # ApproxScalar, which is used to compare individual numbers. All of the 

509 # other Approx classes eventually delegate to this class. The ApproxBase 

510 # class provides some convenient methods and overloads, but isn't really 

511 # essential. 

512 

513 __tracebackhide__ = True 

514 

515 if isinstance(expected, Decimal): 

516 cls = ApproxDecimal 

517 elif isinstance(expected, Number): 

518 cls = ApproxScalar 

519 elif isinstance(expected, Mapping): 

520 cls = ApproxMapping 

521 elif isinstance(expected, Sequence) and not isinstance(expected, STRING_TYPES): 

522 cls = ApproxSequence 

523 elif _is_numpy_array(expected): 

524 cls = ApproxNumpy 

525 else: 

526 raise _non_numeric_type_error(expected, at=None) 

527 

528 return cls(expected, rel, abs, nan_ok)

547def raises(expected_exception, *args, **kwargs): 

548 r""" 

549 Assert that a code block/function call raises ``expected_exception`` 

550 and raise a failure exception otherwise. 

551 

552 :arg message: if specified, provides a custom failure message if the 

553 exception is not raised 

554 :arg match: if specified, asserts that the exception matches a text or regex 

555 

556 This helper produces a ``ExceptionInfo()`` object (see below). 

557 

558 You may use this function as a context manager:: 

559 

560 >>> with raises(ZeroDivisionError): 

561 ... 1/0 

562 

563 .. versionchanged:: 2.10 

564 

565 In the context manager form you may use the keyword argument 

566 ``message`` to specify a custom failure message:: 

567 

568 >>> with raises(ZeroDivisionError, message="Expecting ZeroDivisionError"): 

569 ... pass 

570 Traceback (most recent call last): 

571 ... 

572 Failed: Expecting ZeroDivisionError 

573 

574 .. note:: 

575 

576 When using ``pytest.raises`` as a context manager, it's worthwhile to 

577 note that normal context manager rules apply and that the exception 

578 raised *must* be the final line in the scope of the context manager. 

579 Lines of code after that, within the scope of the context manager will 

580 not be executed. For example:: 

581 

582 >>> value = 15 

583 >>> with raises(ValueError) as exc_info: 

584 ... if value > 10: 

585 ... raise ValueError("value must be <= 10") 

586 ... assert exc_info.type == ValueError # this will not execute 

587 

588 Instead, the following approach must be taken (note the difference in 

589 scope):: 

590 

591 >>> with raises(ValueError) as exc_info: 

592 ... if value > 10: 

593 ... raise ValueError("value must be <= 10") 

594 ... 

595 >>> assert exc_info.type == ValueError 

596 

597 

598 Since version ``3.1`` you can use the keyword argument ``match`` to assert that the 

599 exception matches a text or regex:: 

600 

601 >>> with raises(ValueError, match='must be 0 or None'): 

602 ... raise ValueError("value must be 0 or None") 

603 

604 >>> with raises(ValueError, match=r'must be \d+$'): 

605 ... raise ValueError("value must be 42") 

606 

607 **Legacy forms** 

608 

609 The forms below are fully supported but are discouraged for new code because the 

610 context manager form is regarded as more readable and less error-prone. 

611 

612 It is possible to specify a callable by passing a to-be-called lambda:: 

613 

614 >>> raises(ZeroDivisionError, lambda: 1/0) 

615 <ExceptionInfo ...> 

616 

617 or you can specify an arbitrary callable with arguments:: 

618 

619 >>> def f(x): return 1/x 

620 ... 

621 >>> raises(ZeroDivisionError, f, 0) 

622 <ExceptionInfo ...> 

623 >>> raises(ZeroDivisionError, f, x=0) 

624 <ExceptionInfo ...> 

625 

626 It is also possible to pass a string to be evaluated at runtime:: 

627 

628 >>> raises(ZeroDivisionError, "f(0)") 

629 <ExceptionInfo ...> 

630 

631 The string will be evaluated using the same ``locals()`` and ``globals()`` 

632 at the moment of the ``raises`` call. 

633 

634 .. currentmodule:: _pytest._code 

635 

636 Consult the API of ``excinfo`` objects: :class:`ExceptionInfo`. 

637 

638 .. note:: 

639 Similar to caught exception objects in Python, explicitly clearing 

640 local references to returned ``ExceptionInfo`` objects can 

641 help the Python interpreter speed up its garbage collection. 

642 

643 Clearing those references breaks a reference cycle
